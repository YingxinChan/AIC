# Alembic Migrations Setup — Design Spec

**Date:** 2026-06-23  
**Status:** Approved

## Overview

Add Alembic database migration support to the FastAPI backend using a sync `psycopg2` connection for migrations while keeping `asyncpg` for the application runtime. A single `DATABASE_URL` env var drives both; the sync URL is derived automatically by stripping the `+asyncpg` driver prefix.

## Files Changed

### `backend/requirements.txt`
Add `psycopg2-binary>=2.9` as a migration-only dependency alongside the existing `asyncpg`.

### `backend/core/config.py`
Add a `sync_database_url` computed property that replaces `postgresql+asyncpg://` with `postgresql://` so Alembic can use a standard sync connection without requiring a second env var.

### `backend/models/__init__.py`
Import all three models (`User`, `Trip`, `Activity`) so their classes are registered with `Base.metadata` before Alembic inspects it. Without this, autogenerate sees an empty schema.

### `backend/alembic.ini`
Generated by `alembic init alembic`. The `sqlalchemy.url` field is left blank — the URL is injected at runtime from `settings.sync_database_url` inside `env.py`.

### `backend/alembic/env.py`
Standard sync migration setup:
- Imports `Base` from `models` (which triggers all model imports via `models/__init__.py`)
- Sets `target_metadata = Base.metadata` for autogenerate support
- Reads the database URL from `settings.sync_database_url`

### `backend/alembic/versions/`
One initial migration autogenerated from the three existing models, creating tables: `users`, `trips`, `activities`.

## Running Migrations

All commands run from the `backend/` directory:

```bash
# Apply all pending migrations
alembic upgrade head

# Roll back one migration
alembic downgrade -1

# Generate a new migration after changing models
alembic revision --autogenerate -m "describe what changed"

# Check current migration state
alembic current
```

## Constraints

- **Driver split is intentional:** `asyncpg` stays for app runtime; `psycopg2-binary` is only used by Alembic. This is standard practice and requires no ongoing maintenance.
- **Model imports must be kept up to date:** whenever a new model is added, it must be imported in `models/__init__.py` or Alembic autogenerate will not detect it.
- **Autogenerate is not always perfect:** always review generated migration files before applying, especially for column type changes or PostGIS geometry columns added later.

## Out of Scope

- PostGIS extension setup (no geometry columns yet; add when needed)
- Test database migrations (tests currently use SQLite in-memory)
- CI/CD pipeline integration
